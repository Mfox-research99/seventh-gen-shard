# Contributing a Dissent Record

Thank you for contributing to the Elder Dissent Commons. This guide explains exactly what to submit and how.

---

## Before You Submit

You need three things:

1. **A real session run** — an actual inference call against a model running the Seventh Generation Elder constitution (`config.py` SYSTEM_PROMPT). Not edited, not paraphrased, not written by hand.
2. **The full session log** — the JSON output file from your test script. The dissent text in your record must match the session log verbatim.
3. **A completed dissent record** — a JSON file matching `schema.json`.

---

## Step-by-Step

### 1. Run a session

Use the canonical scenarios from `config.py`. The simplest path:

```bash
# Clone the repo
git clone https://github.com/Mfox-research99/seventh-shard.git
cd seventh-shard

# Edit test_anubis_suite.py (or test_qwen_suite.py or test_full.py)
# to point MODELS at your model path

# Run
python test_anubis_suite.py
# Output saved to logs/anubis_suite_TIMESTAMP.json
```

You can use any model you have locally. Point the `MODELS` list at your model path and run. The test scripts handle the rest.

### 2. Identify the dissent

Look at the session log output. A dissent is any response where:
- The Elder refuses or escalates when the scenario target is "proceed"
- The Elder proceeds when the scenario target is "refuse" or "escalate"
- The Elder reaches a verdict that diverges meaningfully from other Elders on the same scenario
- The Elder's reasoning names a harm pattern or forward condition not captured in prior records

Single-model sessions are valid. You don't need a multi-model council to contribute a dissent.

### 3. Compute the session log hash

```bash
shasum -a 256 logs/your_session_log.json
```

Save this hash — it goes in the dissent record and is verified by CI.

### 4. Complete the dissent record

Copy `examples/dissent_example.json` as a starting point. Fill in every required field. Pay special attention to `forward_conditions` — these are the most important field. Be specific. Name the exact future condition the Elder's reasoning implies.

Save your record to `dissents/commons/your_session_id.json`.

### 5. Submit a pull request

Your PR must include:
- The dissent record JSON in `dissents/commons/`
- The session log JSON in `dissents/logs/` (named `session_SESSIONID.json`)
- A brief PR description: which model, which scenario, what the dissent is about

CI will automatically verify:
- The dissent record matches `schema.json`
- The session log hash matches the hash in the record
- Required fields are present and non-empty
- The dissent text appears verbatim in the session log

**PRs that fail CI will not be reviewed until they pass.**

### 6. Human review

A maintainer will read the dissent, check that it represents genuine Elder reasoning (not a hallucination, jailbreak, or off-topic response), and merge if it meets the bar.

Review typically takes 3–7 days.

## What We Are Looking For

**Strong submissions:**
- The Elder names specific future conditions that are checkable against reality
- The reasoning engages with the Seventh Generation harm taxonomy (Soul.md Article IX)
- The dissent is from a model or architecture not yet represented in the commons
- The dissent is on a scenario where prior records show the opposite verdict — genuine divergence is valuable

**Submissions that will be declined:**
- Records where the dissent text does not match the session log
- Records where the model was clearly jailbroken or manipulated to produce the response
- Records with vague forward conditions ("things might go wrong") rather than specific ones ("the 90-day sunset clause will be removed at first renewal under budget pressure")
- Duplicate dissents that add no new reasoning to an existing record on the same scenario

---

## Integrity Policy

**All submissions are verified against session logs. Fabricated dissents — records not generated by an actual model run — are not permitted.**

Fabrication includes: writing the dissent by hand, editing model output to strengthen the reasoning, or submitting a session log that does not match the actual run.

Contributors found submitting fabricated records will be permanently banned from contribution.

This is a research commons, not a debate forum. The value of the record depends entirely on its integrity.

---

## Model Requirements

There are none beyond the Elder constitution. Any model that can run the SYSTEM_PROMPT from `config.py` is eligible. This includes:

- Any HuggingFace model run via mlx-lm, llama.cpp, vllm, ollama, or transformers
- Models with or without the Seventh Shard LoRA adapter
- Models of any size (we want to see how small models reason as Elders)
- Cloud-hosted models if you can capture the full session output

Document your model source, version, and quantization accurately. Reproducibility matters.
